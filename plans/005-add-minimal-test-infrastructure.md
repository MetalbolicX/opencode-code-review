## Why this matters

This repo is mostly deterministic string generation (prompt builders), config merging, and light plugin wiring. The highest-value protection comes from unit tests around prompt builders, dimension filtering, and config merge precedence. A tiny plugin-registration smoke test covers the public contract. Full E2E against the real OpenCode host is overkill at this stage ŌĆö it would be expensive, brittle, and low-ROI until the local unit/integration baseline exists. This plan establishes that baseline.

### Step 2: Write unit tests for `src/agent.ts` prompt builders

Create `src/agent.test.ts`.

Test cases to cover:
- `buildAgentPrompt` returns the single-review prompt when `config.parallel` is `false`.
- `buildAgentPrompt` returns the orchestrator prompt when `config.parallel` is `true`.
- `buildAgentPrompt` produces different output for `language: "zh"` vs `language: "en"`.
- `buildFixerPrompt` returns non-empty output for both `zh` and `en`.
- `buildTogglePrompt` returns non-empty output for both `zh` and `en`.
- `buildCustomRules` (via `buildAgentPrompt`) injects custom rules into the prompt when `custom_rules` is non-empty.
- `buildCustomRules` does not inject a Custom Rules section when `custom_rules` is empty.

Assertion style: check for known substrings (e.g., prompt contains "review_changes" tool reference, contains "review:fixer", contains the dimension labels). Do NOT use snapshot tests for full prompts ŌĆö wording changes legitimately, and snapshots create noise. Use targeted substring assertions instead.

**Verify**: `pnpm test src/agent.test.ts` exits 0 with all cases passing.

### Step 3: Write unit tests for `src/dimensions/index.ts`

Create `src/dimensions/index.test.ts`.

Test cases to cover:
- `getDimensionPrompts` filters out unknown dimension names (e.g., `"not-a-dimension"` produces no entry).
- `getDimensionPrompts` deduplicates repeated dimension names.
- `getDimensionPrompts` returns the correct `agentName` format (`review:dim-<name>`).
- `getDimensionPrompts` returns non-empty `prompt` strings for known dimensions.
- `getDimensionPrompts` selects zh vs en content based on `config.language`.
- `getDimensionPrompts` with all 5 default dimensions returns exactly 5 entries.

**Verify**: `pnpm test src/dimensions/index.test.ts` exits 0 with all cases passing.

### Step 4: Write unit tests for `src/config.ts` merge logic

Create `src/config.test.ts`.

This is the trickiest unit because `loadConfig` reads from the filesystem (`~/.config/opencode/review.json` and `<projectDir>/.opencode/review.json`). To test it cleanly, mock `node:fs/promises` `readFile`.

Test cases to cover:
- Returns `DEFAULT_CONFIG` when no config files exist (both reads fail/return null).
- Global config overrides defaults (e.g., global sets `language: "en"`, result has `language: "en"`).
- Project config overrides global config (e.g., global sets `"en"`, project sets `"zh"`, result is `"zh"`).
- Nested `trigger` merges correctly: global sets `trigger.auto_on_idle: true`, project sets `trigger.cooldown_seconds: 60`, result has both.
- Malformed JSON in a config file falls back safely to defaults (does not throw).
- Partial config (only some fields) merges without losing unrelated default fields.

Mocking approach: use `vi.mock("node:fs/promises", ...)` to control `readFile` return values per test.

**Verify**: `pnpm test src/config.test.ts` exits 0 with all cases passing.

### Step 5: Write one smoke integration test for plugin registration

Create `src/index.test.ts`.

This is a thin smoke test only ŌĆö do NOT build a full fake OpenCode runtime.

Test cases to cover:
- The default export is a function (the plugin factory).
- Calling the plugin factory with a minimal mock context returns an object with `config`, `tool`, and `event` keys.
- The `tool` object contains `review_changes` and `toggle_auto_review`.
- The `config` function, when called with an empty config object, registers the `review` and `review:auto` commands.

Mock approach: construct a fake context object `{ project: "", client: {}, $: vi.fn(), directory: process.cwd(), worktree: "" }`. The `client` needs a `session.promptAsync` stub (never called in this test). The `config` function receives a mutable `openCodeConfig` object ŌĆö assert that it populates `openCodeConfig.agent["review"]` and `openCodeConfig.command["review"]`.

If the OpenCode plugin contract is too complex to mock cheaply, STOP and report ŌĆö do not build an elaborate harness for one smoke test.

**Verify**: `pnpm test src/index.test.ts` exits 0 with all cases passing.
